# Daily Status - 2026-02-23

## Accomplishments

- **[MGMT-23031](https://issues.redhat.com/browse/MGMT-23031) — Changed default provisioning provider from EDA to AAP**
  - Updated `cmd/main.go` and `config/samples/osac-config-secret.yaml` to default to AAP when `OSAC_PROVISIONING_PROVIDER` is unset
  - Rebased main on upstream; resolved cherry-pick conflict for already-merged commit (MGMT-22639 finalizer)

- **[MGMT-23103](https://issues.redhat.com/browse/MGMT-23103) — ComputeInstance API Enhancement: E2E Validation**
  - Fixed re-provision logic: added `needsProvisionJob()` to handle spec changes (e.g. runStrategy) triggering new provision jobs
  - Fixed E2E test to use `spec.runStrategy` instead of deprecated `spec.running`
  - Found and fixed phase regression bug: phase no longer drops to `Starting` on re-provision of a running instance
  - Found and fixed re-provision timeout bug: playbook now skips "Wait for VM ready" when `runStrategy=Halted` (PR [osac-templates#8](https://github.com/osac-project/osac-templates/pull/8))
  - Removed VMI-level checks from E2E test; VM-level `printableStatus` is sufficient
  - Re-vendored osac.templates in osac-aap (PR [osac-aap#205](https://github.com/osac-project/osac-aap/pull/205) merged, EE image rebuilt)
  - Found and fixed deprovision spec validation error: switched finalizer role from server-side apply to JSON merge patch (PR [osac-aap#206](https://github.com/osac-project/osac-aap/pull/206))
  - Found and fixed tenant-not-ready requeue bug: controller now properly requeues instead of signaling "done"
  - Found and fixed tenant GC race condition: operator now detects terminating tenants and requeues instead of updating them
  - **E2E test passed** — all 7 checks passed (create, runStrategy Halted/Always, immutability enforcement, cleanup)
  - Started removal of dead `ensureBackwardCompatibility()` code (in progress)

- **[MGMT-23103](https://issues.redhat.com/browse/MGMT-23103) — Added `Updating` phase to ComputeInstance API**
  - Added `COMPUTE_INSTANCE_STATE_UPDATING = 5` to fulfillment-api proto (PR [fulfillment-api#61](https://github.com/osac-project/fulfillment-api/pull/61), review pending)
  - Added same enum to fulfillment-service private API proto (PR [fulfillment-service#306](https://github.com/osac-project/fulfillment-service/pull/306))
  - Added `Updating` phase to osac-operator with state machine refactor and comprehensive tests (PR [osac-operator#114](https://github.com/osac-project/osac-operator/pull/114), 110 tests pass)
  - Pending: BSR publication and replacing proto state literal with named constant

## Risks & Challenges

- BSR publication of fulfillment-api and private-api protos is required before the operator can use the named `Updating` constant instead of a magic number
- The `Updating` phase addition spans three repos (fulfillment-api, fulfillment-service, osac-operator) with coordinated merge/publish dependencies
- Follow-up needed for a proper `Updating` phase in the CRD + fulfillment API beyond the current mitigation

## Key Effort

Drove the [MGMT-23103](https://issues.redhat.com/browse/MGMT-23103) ComputeInstance API enhancement to E2E validation, discovering and fixing six bugs along the way (re-provision logic, phase regression, playbook timeout, deprovision validation, requeue handling, and tenant GC race), and began adding the `Updating` phase across the fulfillment stack.
