# Daily Status - 2026-03-15

- Fixed block-deletion-on-failure behavior and UTC timestamp handling for cluster orders
- Implemented config version tracking and duplicate trigger guard for ClusterOrder and HostPool resources, ensuring spec changes trigger re-provisioning while preventing duplicate jobs from stale cache reads
- Added comprehensive unit tests covering provision triggering logic, config version handling, and reconciliation behavior
- Deployed to vmaas-dev and ran end-to-end tests — verified ClusterOrder and HostPool provisioning, job polling, and config version tracking all working correctly
- Discovered a critical bug during E2E testing: failed provision jobs cause an infinite retry loop because the reconciled config version is never set on failure, causing perpetual re-triggering — this affects all three resource types (ClusterOrder, HostPool, ComputeInstance)
- Drafted a proposed fix with multiple approaches (annotation-based retry, spec field retry, exponential backoff) for discussion with the team
- Added integration tests for ClusterOrder (10 tests) and HostPool (8 tests) covering the full provision lifecycle, failure handling, stale cache guard, deprovisioning, and config version behavior
